# Spec — Item 13: Freetext Describe Box

**Phase**: B (Full question flow) · **Status**: Draft, awaiting approval

## Goal

Item 12 introduced the freetext step as a structural placeholder — an empty `<textarea>` between Q3 and submit. Item 13 makes that step do its product job: when the user arrives at the freetext step, the textarea is **pre-filled with a deterministic natural-language summary of their answers so far** (e.g., "A comforting regular meal, for 2, under ₹350 each."). The user can accept it, edit it, or clear it entirely. Whatever ends up in the textarea is sent on the wire as `answers.freetext` and read by the model as the **primary intent** — the override semantics from product spec §4.3.

The summary is generated by a single helper, `buildIntentSummary(answers, context)`, that lives in `shared/` per the build plan's "Reusable utilities" list. Item 13 wires the helper into the freetext pre-fill on the frontend. The `context` argument carries `avgOrderValue` from the seed profile so the budget clause can surface the actual rupee figure (e.g., "under ₹350") rather than the abstract word "budget" — resolved during plan review against spec §4.3. The prompt builder change is deliberately scoped to **zero behaviour change** in this item — `backend/src/prompt.ts` already surfaces `Freetext: …` in the dynamic context block (added in Item 06), and the existing line continues to flow whatever the user typed (including the auto-summary) straight to the model. No new BE dependency on `buildIntentSummary` lands in this item; it is shared-side so a future item (refinement loop in Item 18, persona swap in Item 20) can call the same generator without re-implementing the grammar.

What this item is **not**: it does not change the wire contract (no new fields on `RecommendAnswers`, no Zod schema change, no BE schema change), does not change the static cached system-prompt prefix (the prompt-caching invariant holds), does not animate the prefill or its replacement, and does not introduce a "regenerate suggestion" affordance. It also does not introduce a "primary intent vs supplementary signals" weighting on the BE — the existing `Freetext:` line in the dynamic block, combined with the §5 ranking algorithm's "Dish description match (10 pts) — applies only when the user supplied freetext", is the override mechanism. Pre-filling the textarea means the model now sees freetext on every request (rather than rarely), but the prompt structure is unchanged.

## Deliverables

- `shared/intentSummary.ts` — **new**. Pure, deterministic NL summariser:
  - One exported function `buildIntentSummary(answers: IntentInputs, context: IntentContext): string` where `IntentInputs = Pick<RecommendAnswers, 'q1' | 'q2' | 'q3' | 'partySize'>` and `IntentContext = { avgOrderValue: number }`. The context is positional-second so callers must supply the rupee figure explicitly — the helper never reaches into profile state itself.
  - Output is a single English sentence ending in a period. Always non-empty (Q1 is required on `RecommendAnswers`).
  - Same input → same output across calls. No randomness, no time-dependent phrasing, no `Math.random`, no `new Date()` reads. Pure function of its argument.
  - No I/O, no localStorage, no `console`, no React imports. The file imports only the types it needs from `shared/types.ts`.
  - Grammar (locked at this granularity; exact wording finalised at implementation per Open Questions):
    - **Base clause** combines Q1 with Q2 (Q2-aware variant when both set). When only Q1 is set, Q1's own wording carries.
    - **Q3 modifier clauses** append in a fixed order — `veg-only` → `fast-delivery` → `high-rated` → `budget` — each as a comma-separated trailing fragment.
    - **partySize** attaches as part of the `budget` clause when both are present (e.g., "for 4, under ₹350 each"). Without `budget`, partySize is not surfaced (it's only meaningful when budget is selected — per Item 11's wire contract).
    - Always ends with a single period. No double-period, no trailing comma, no leading or trailing whitespace.
  - See the Tech choices table for the full output-string grammar and worked examples — these strings are part of the contract.

- `frontend/src/screens/Questions.tsx` — **modified**. Two small additions to the existing freetext step wiring:
  1. **New import**: `import { buildIntentSummary } from "@shared/intentSummary";`.
  2. **New effect**: pre-fill the textarea exactly once, when the user first navigates into the freetext step. Concretely, a `useEffect` with `[step]` in the dependency array and a one-shot `didPrefillFreetextRef` guard (mirrors the existing `didCountSkipRef` pattern in the same file). The effect:
     - Fires only when `step === 'freetext'`.
     - Bails out if `didPrefillFreetextRef.current === true`.
     - Bails out if `freetext.length > 0` (defensive — the user can't realistically have typed before the effect runs in the same render, but the guard makes the intent explicit and protects against future re-orderings).
     - Builds the `IntentInputs` object from the current `hunger`, `mealType`, `q3`, and `partySize` state. **Skips `partySize` when `q3` does not contain `'budget'`** (mirrors the existing wire-omission rule from Item 11's submit handler).
     - Sets `freetext` to `buildIntentSummary(inputs)`.
     - Sets `didPrefillFreetextRef.current = true`.
  3. **No change to the `<textarea>` element itself** — `value`, `onChange`, `maxLength`, `placeholder`, and class names from Item 12 stay byte-for-byte identical. The textarea is still fully controlled by `freetext` state.
  4. **No change to the submit handler** — `if (freetext.trim().length > 0) answers.freetext = freetext.trim();` from Item 12 already does the right thing: the pre-fill is non-empty, so it gets sent; if the user clears the textarea, the field is omitted; whitespace-only stays omitted.

- `frontend/src/screens/Questions.tsx` — **placeholder + helper-copy updates** in the freetext-step body (small):
  - The heading copy may change from Item 12's placeholder `"Anything specific?"` to a phrasing that better matches the new "review and refine" UX — working assumption: `"How does this sound?"` Confirm against spec §4.3 at implementation time (Open Question).
  - The helper-text copy may change from Item 12's placeholder `"Add cravings, constraints, or anything else worth knowing. Optional."` to something like `"Edit to refine. We'll use this as your primary intent."` Again, §4.3 wins.
  - The `placeholder=` attribute on the textarea (`"Any specific cravings or constraints?"`) becomes mostly invisible because the textarea is now pre-filled on arrival. Leave the placeholder text as-is — it still shows if the user clears the textarea and re-enters it, which is the only realistic visible-placeholder case.

That is the full scope.

| **Output grammar — Q3 `high-rated` modifier** | `, from top-rated places`. | Reads naturally even at the end of the sentence. Avoids the awkward `"high-rated"` adjective. |

| **Punctuation — comma separation between modifier clauses** | Each modifier clause begins with `, ` (comma + space). Assembled as `base + modifier1 + modifier2 + … + '.'`. | One assembly path: `[base, ...selectedModifiers].join('') + '.'` (where modifiers already carry their leading `, `). Equivalently, `base + modifiers.map(m => ', ' + m.body).join('') + '.'`. |
| **What happens when no modifiers apply** | Output is just the base clause + period. Example: `{q1:'regular-meal'}` → `"A regular meal."`. | The base clause alone is a complete sentence. No "Something..." fallback when Q2/Q3 are empty. |
| **What happens when Q3 contains `budget` but no partySize is provided** | Modifier reads `, under ₹<avgOrderValue>` (no `for N`, no `each`). | The IntentInputs caller (Questions.tsx) is responsible for only passing `partySize` when Q3 includes `'budget'` (matches the submit-handler omission rule). | **What happens when `partySize` is passed without `budget`** | Helper silently ignores `partySize`. Output unchanged. | Defence-in-depth. | **What happens when q3 is an empty array** | Empty array treated as "no modifiers". Same as `q3: undefined`. | The wire contract distinguishes "absent" from "empty" but for the summariser both mean "no constraints". |
| **Pre-fill trigger — when does the effect fire** | Once, when `step` first becomes `'freetext'` in a given component-mount lifetime. Subsequent step changes (back-and-forth) do not re-fire. | The simplest mental model: user reaches freetext step → sees the summary → owns the textarea from that point on. | **Pre-fill trigger — exact React mechanism** | `useEffect(() => { … }, [step])`, guarded by `useRef<boolean>(false)` initialised to `false`, flipped to `true` after the first prefill runs. ## Implementation notes

- **`shared/intentSummary.ts` shape**:
  ```ts
  import type { HungerLevel, MealType, Q3Constraint, RecommendAnswers } from "./types";

  export type IntentInputs = Pick<RecommendAnswers, "q1" | "q2" | "q3" | "partySize">;
  export type IntentContext = { avgOrderValue: number };

  const BASE: Record<HungerLevel, string> = {
    "light-snack": "Something light to snack on",
    "regular-meal": "A regular meal",
    "very-hungry": "Something filling — I'm very hungry",
  };

  const BASE_WITH_Q2: Record<HungerLevel, Record<MealType, string>> = {
    "light-snack": {
      "comfort-favourite": "A familiar light snack",
      "healthy": "A light healthy snack",
      "indulgent": "A small indulgent treat",
      "surprise-me": "Surprise me with a light snack",
    },
    "regular-meal": {
      "comfort-favourite": "A comforting regular meal",
      "healthy": "A healthy regular meal",
      "indulgent": "An indulgent regular meal",
      "surprise-me": "Surprise me with a regular meal",
    },
    "very-hungry": {
      "comfort-favourite": "A big comforting meal — I'm very hungry",
      "healthy": "A big healthy meal — I'm very hungry",
      "indulgent": "A big indulgent meal — I'm very hungry",
      "surprise-me": "Surprise me with a big meal — I'm very hungry",
    },
  };

  // Canonical order, regardless of user-selection order.
  const MODIFIER_ORDER: Q3Constraint[] = ["veg-only", "fast-delivery", "high-rated", "budget"];

  function formatRupees(n: number): string {
    return `₹${Math.round(n)}`;
  }

  function modifierFor(
    chip: Q3Constraint,
    partySize: number | undefined,
    avgOrderValue: number,
  ): string {
    switch (chip) {
      case "veg-only": return "veg";
      case "fast-delivery": return "delivered fast";
      case "high-rated": return "from top-rated places";
      case "budget":
        return partySize !== undefined
          ? `for ${partySize}, under ${formatRupees(avgOrderValue)} each`
          : `under ${formatRupees(avgOrderValue)}`;
      default: {
        const _exhaustive: never = chip;
        return _exhaustive;
      }
    }
  }

  export function buildIntentSummary(
    answers: IntentInputs,
    context: IntentContext,
  ): string {
    const base = answers.q2 ? BASE_WITH_Q2[answers.q1][answers.q2] : BASE[answers.q1];
    const q3 = answers.q3 ?? [];
    // partySize is only meaningful with budget — defence-in-depth ignore otherwise.
    const partySize = q3.includes("budget") ? answers.partySize : undefined;
    const modifiers = MODIFIER_ORDER
      .filter((m) => q3.includes(m))
      .map((m) => modifierFor(m, partySize, context.avgOrderValue));
    return [base, ...modifiers].join(", ") + ".";
  }
  ```
  No exported helpers other than `buildIntentSummary`, `IntentInputs`, and `IntentContext`. The `default` branch on `modifierFor` is an unreachable exhaustiveness guard — adding a new `Q3Constraint` literal to `shared/types.ts` will fail compilation here until a case is added.

- **`Questions.tsx` effect**:
  ```ts
  import { buildIntentSummary, type IntentInputs } from "@shared/intentSummary";

  // …inside the component, alongside the existing didCountSkipRef:
  const didPrefillFreetextRef = useRef(false);

  // One-shot prefill: when the user first reaches the freetext step, populate
  // the textarea with the deterministic NL summary of their answers. Subsequent
  // back/forward navigation preserves any edits the user has made.
  useEffect(() => {
    if (step !== "freetext") return;
    if (didPrefillFreetextRef.current) return;
    if (freetext.length > 0) return;          // defensive — pre-existing state
    if (!hunger) return;                       // can't realistically happen — Q1's Next is disabled without hunger
    const inputs: IntentInputs = {
      q1: hunger,
      ...(mealType ? { q2: mealType } : {}),
      ...(q3.length > 0 ? { q3 } : {}),
      ...(q3.includes("budget") ? { partySize } : {}),
    };
    setFreetext(
      buildIntentSummary(inputs, { avgOrderValue: profile.avgOrderValue }),
    );
    didPrefillFreetextRef.current = true;
    // eslint-disable-next-line react-hooks/exhaustive-deps -- reason: one-shot prefill on step transition; depending on hunger/mealType/q3/partySize would re-fire on every chip toggle and clobber user edits.
  }, [step]);
  ```
  The trailing `-- reason:` note on the eslint-disable line documents why the dep array is intentionally narrow.

- **Why the inputs object uses conditional spread** — `Pick<RecommendAnswers, …>` carries optional fields. Passing `{q1, q2: null, q3: [], partySize: 2}` would force the helper to treat `null` as "set" (it's not). Spreading only when set produces a clean `IntentInputs` matching the contract. The helper is robust against the alternative shape (empty arrays, undefined fields), but the call site should still send the cleanest possible object — same omission-rule discipline as the submit handler.

- **Why the Q1 base table includes `light-snack × surprise-me` as `"Surprise me with a light snack"`** — Q2's `surprise-me` is a directive ("surprise me") rather than a descriptor. Composing it with Q1 as `"A surprise-me light snack"` would be ungrammatical. The lookup table handles all 12 combinations as fixed strings to avoid the model trying to compose ungrammatical sentences.

- **Why the helper uses `Pick<RecommendAnswers, …>` rather than a fresh interface** — `RecommendAnswers` already exists in `shared/types.ts`. A separate `IntentInputs` interface would drift over time. `Pick` keeps the helper's input type pinned to the canonical wire type minus the field it can't accept (`freetext`, which would be circular).

- **Editing flow** — after the prefill, the textarea is fully owned by the user. `onChange={(e) => setFreetext(e.target.value)}` from Item 12 already handles every keystroke; the ref guard prevents the effect from clobbering edits if the user pops back to Q3 and returns.

- **Clear-and-retype flow** — if the user wipes the pre-fill (`Cmd+A`, Delete), they see the placeholder `"Any specific cravings or constraints?"` from Item 12. They can type a fresh sentence; that becomes `answers.freetext`. The pre-fill does not re-populate, even if they navigate back and forward — the ref stays `true`.

- **Submit with whitespace-only or empty textarea** — Item 12's `if (freetext.trim().length > 0) answers.freetext = freetext.trim();` rule continues to hold. If the user clears the box (intentional skip), `answers.freetext` is omitted. The prompt builder's `Freetext: —` line then runs. This is acceptable: a deliberate clear is the user saying "no extra intent".

- **No new copy on the "review" framing** — heading and helper-text changes are documented as working assumptions; the §4.3 source-of-truth wins. The implementer should re-read §4.3 (re-read `nudge_spec.docx` §4.3) at plan-review time and propose final copy. If §4.3 explicitly names the heading, use that verbatim.

- **No JSDoc on `buildIntentSummary`** — the function name + types are self-documenting. The grammar lives in this spec and in the inline `BASE` / `BASE_WITH_Q2` tables, which are the file's documentation by structure.

- **No tests in this item** — added via `/test-feature 13-freetext-box` after implementation.
